- # PR Descriptionを更新

PR #$ARGUMENTS のDescriptionを以下の形式で自動生成して更新してください。
`#$ARGUMENTS` が未指定の場合は現在のブランチのPRが存在するかを確認してください。
PRが存在しない場合は、ユーザーに確認してください。

もし、PRタイトルが「ブランチ名 タイトル」のままだったら、タイトルの部分を適切に書き換えてください。

## 手順

1. `gh pr view $ARGUMENTS` でPRの情報を取得
2. `gh pr diff $ARGUMENTS` でコミット内容の差分を取得
3. 差分を分析して、以下の形式でDescriptionを作成

## テンプレート

```markdown
## 概要

[なぜこの変更が必要か、どんな問題を解決するかを1-2文で記述]

## 変更内容

[主要な変更点を箇条書きで記述。レビュアーが差分を見る前に全体像を把握できるように]

## レビューポイント

[以下の観点で記述]

- 特に確認してほしい箇所
- 実装で迷った点や相談事項
- 意図的にスコープ外にした点（あれば）

## 確認済み

[PR作成者が確認した内容を箇条書きで記述]

- 確認した画面やパス
- 確認したケース(正常系、エラー時など)

## スクリーンショット

[UI変更がある場合のみ記載。Before/Afterがあると理想的]

<!-- 以下は該当する場合のみ記載 -->

## 影響範囲

[他機能への影響がある場合に明示]

## 破壊的変更

[API変更や後方互換性がない場合の警告]

## デプロイ時の注意

[環境変数追加、マイグレーション、依存関係の更新など]

---

🤖 Generated with [Claude Code](https://claude.com/claude-code)
```

4. `gh pr edit $ARGUMENTS --body "..."` でDescriptionを更新

## 生成時の注意

- **概要**: 「〇〇対応」ではなく、なぜその変更が必要かを明記
- **変更内容**: ファイル一覧ではなく、機能単位で何が変わったかを記述
- **レビューポイント**: 差分から複雑な箇所や重要な判断を抽出して提示
- **確認済み**: 差分から推測される確認項目を記述（画面パス、正常系/異常系など）
- **スクリーンショット**: UI変更がない場合はセクションごと省略
- **影響範囲/破壊的変更/デプロイ時の注意**: 該当する場合のみ記載。該当しない場合はセクションごと省略
